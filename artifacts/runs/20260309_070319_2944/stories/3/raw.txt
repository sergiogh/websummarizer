(function () {

  function isMobile() {
    return window.innerWidth <= MOBILE_BREAK;

  // Centralized close logic
  function closeAll() {
    desktopItems.forEach(item => {
      item.classList.remove('is-open');
      const l = item.querySelector('.bcl-nav-list__item-link');
      const d = item.querySelector('.main-nav__dropdown');
      if (l) l.setAttribute('aria-expanded', 'false');
      if (d) d.setAttribute('aria-hidden', 'true');
    });

    document.documentElement.classList.remove('no-scroll');
    document.querySelectorAll('header.experiencefragment')
      .forEach(h => h.classList.remove('fixed--header'));

    const overlay = document.querySelector('.nav-overlay');
    if (overlay) overlay.classList.remove('active');
  }

  // Document click handler
  function onDocumentClickClose(e) {
    if (e && e.target) {
      const allDropdowns = Array.from(document.querySelectorAll('.main-nav__dropdown'));
      if (allDropdowns.some(dd => dd && dd.contains(e.target))) {
        return; // ignore clicks inside dropdown
      }
    }

    if (docListenerAttached) {
      document.removeEventListener('click', onDocumentClickClose, true);
      docListenerAttached = false;
    }

    closeAll();
  }

  function attachDocListenerOnce() {
    if (docListenerAttached) return;
    document.addEventListener('click', onDocumentClickClose, true);
    docListenerAttached = true;
  }

  function setDropdownHeight(){
      let header = document.querySelector('.bcl-header');
      let headerHeight = header ? header.offsetHeight : 0;

      // dropdown height
      let dropdownHeight = headerHeight;

      // eyebrow height (if exists)
      let eyebrow = document.querySelector('.bcl-eyebrow');
      if (eyebrow) {
          dropdownHeight += eyebrow.offsetHeight;
      }
      if(isMobile()){
          dropdownHeight = -dropdownHeight+5;
      }else{
          dropdownHeight = dropdownHeight;
      }

        document.querySelectorAll('.main-nav__dropdown').forEach(dropdown => {
          if(isMobile()){
              dropdown.style.top =0;
              dropdown.style.height = '100vh';
              dropdown.style.overflowY = 'auto';
          }else{
              dropdown.style.top = dropdownHeight + 'px';
              dropdown.style.height = 'auto';
          }
      });

      document.querySelectorAll('.nav-overlay').forEach(dropdown => {
          dropdown.style.top = dropdownHeight + 'px';
      });
  }

  desktopItems.forEach(item => {
    const link = item.querySelector('.bcl-nav-list__item-link');
    const dropdown = item.querySelector('.main-nav__dropdown');
    if (!link || !dropdown) return;
    
    link.addEventListener('click', (e) => {
      if (isMobile()) return;
      e.preventDefault();

      setDropdownHeight();
      const wasOpen = item.classList.contains('is-open');

      desktopItems.forEach(i => {
        i.classList.remove('is-open');
        const l = i.querySelector('.bcl-nav-list__item-link');
        const d = i.querySelector('.main-nav__dropdown');

      document.querySelectorAll('header.experiencefragment')

      document.documentElement.classList.remove('no-scroll');

      if (overlay) overlay.classList.remove('active');

      // If it was closed, open it now
      if (!wasOpen) {

        if (overlay) overlay.classList.add('active');

      } else {
        // Already closed
        if (docListenerAttached) {
          document.removeEventListener('click', onDocumentClickClose, true);
          docListenerAttached = false;
        }
      }
    });

    // Keyboard accessibility
    link.addEventListener('keydown', (e) => {
      if (isMobile()) return;
      if (e.key === 'Enter' || e.key === ' ') {
        e.preventDefault();
        link.click();
      }
    });

              // activate selected
              tab.classList.add('cmp-tabs__tab--active');
              const panel = document.getElementById(panelId);
              if (panel) panel.classList.add('cmp-tabs__tabpanel--active');
          });

      if (tabLinks.length > 0) {

      }
  }

  if (!isMobile()) {

  // ESC closes all

    if (e.key === 'Escape') {
      if (docListenerAttached) {
        document.removeEventListener('click', onDocumentClickClose, true);
        docListenerAttached = false;
      }
      closeAll();
    }
  });

<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>Throughout our <a href="#previous-posts" target="_self">series of blog posts</a> on post-quantum cryptography (PQC), we have focused almost exclusively on the Transport Layer Security (TLS) ecosystem in an HTTP context. Addressing the “harvest now, decrypt later” threat in the CDN context was an obvious high priority for Akamai, and we are pleased to have delivered PQC in our TLS 1.3 handshakes for both <a href="https://techdocs.akamai.com/property-mgr/docs/pqc-client-to-edge" target="_blank" rel="nofollow noreferrer">client-to-Akamai</a> and <a href="https://techdocs.akamai.com/property-mgr/docs/pqc-origin" target="_blank" rel="nofollow noreferrer">Akamai-to-origin</a> connections.</p>
<p>Over the coming weeks, these features are being rolled out everywhere as on-by-default for all <a href="https://techdocs.akamai.com/edge-ip-binding/docs/eipb-enhanced-tls-https" target="_blank" rel="nofollow noreferrer">Enhanced TLS</a> customers.</p>
<p>Although this is in line with the rest of the industry's prioritizations, the quantum threat affects all areas of cryptography, not just TLS. In this blog post, we'll explore some of the other protocols and services that will warrant our attention when it comes to making them quantum safe.</p>

</div>

    

</div>
<div class="title padding__bottom-three title_sm aem-GridColumn aem-GridColumn--default--12">
<div class="cmp-title" id="title-87fb5e90fb">
    
      <h2 class="cmp-title__text" data-context="title">Algorithms that are subject to quantum threats <br />
</h2>
    
</div>

    


</div>
<div class="text bullet__xs padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>Practically speaking, we are concerned about any software that uses the algorithms known to be subject to quantum threats; that is, any use of asymmetric key cryptography (e.g., cryptography that uses RSA or Diffie–Hellman key exchanges), as well as digital signatures (e.g., Elliptic Curve Digital Signature Algorithm [ECDSA] are of primary concern).</p>
<p>It's worth noting that although TLS covers the biggest surface area within the HTTP ecosystem, it is also used by many other applications, including:</p>
<ul>
<li><p>The Simple Mail Transfer Protocol (SMTP) with STARTTLS or via implicit TLS submission on port 465 per <a href="https://www.rfc-editor.org/rfc/rfc8314" target="_blank" rel="nofollow noreferrer">RFC8314</a></p>
</li>
<li><p>The Internet Message Access Protocol (IMAP) or Post Office Protocol (POP) over TLS</p>
</li>
<li><p>The Lightweight Directory Access Protocol (LDAP) over TLS</p>
</li>
<li><p>TLS-based virtual private networks (VPN)</p>
</li>
<li><p>Modern network segmentation services like <a href="/glossary/what-is-ztna" target="_self">Akamai's Zero Trust Network Access (ZTNA)</a></p>
</li>
<li><p>The Extensible Messaging and Presence Protocol (XMPP) used for some instant messaging services</p>
</li>
</ul>
<p>Since all of these services are using TLS for traffic encryption and endpoint authentication, upgrading these services to use the Post-Quantum/Traditional (PQ/T) hybrid key exchange algorithm we discussed at length in <a href="/blog/trends/building-quantum-safe-internet-ietf-plan-tls" target="_self">previous</a> <a href="/blog/security/post-quantum-cryptography-implementation-considerations-tls" target="_self">blog</a> <a href="/blog/security/guide-international-post-quantum-cryptography-standards" target="_self">posts</a> (i.e., <span class="code">X25519MLKEM768</span>) will likewise help protect them against the quantum threat.</p>
<p>However, other applications and services use different protocols and will need to likewise implement PQ/T hybrid key exchange algorithms and begin to use post-quantum signature algorithms.</p>

</div>

    

</div>
<div class="title padding__bottom-three title_sm aem-GridColumn aem-GridColumn--default--12">
<div class="cmp-title" id="title-36b99c669a">
    
      <h2 class="cmp-title__text" data-context="title">Secure Shell (SSH)<br />
</h2>
    
</div>

    


</div>
<div class="text bullet__xs padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>A primary example here is the ubiquitous SSH protocol. As you know, SSH also uses public-key cryptography to perform authentication, and as such needs to be configured to use the right key exchange algorithms to provide quantum safety.</p>
<p>The most widely used implementation of SSH, <a href="https://www.openssh.org/" target="_blank" rel="nofollow noreferrer">OpenSSH</a>, has had support for the Streamlined NTRU Prime (SNTRUP) algorithm in a PQ/T hybrid configuration since 2021. In 2024, <a href="https://csrc.nist.gov/pubs/fips/203/final" target="_blank" rel="nofollow noreferrer">FIPS 203</a> Module-Lattice-Based Key-Encapsulation Mechanism (ML-KEM) combined with X25519 ECDH (as <a href="https://datatracker.ietf.org/doc/draft-ietf-sshm-mlkem-hybrid-kex/" target="_blank" rel="nofollow noreferrer">defined by the Internet Engineering Task Force [IETF]</a>) was added, which became the default key exchange algorithm starting with version 9.9 — meaning that a simple upgrade across your operating systems will protect you today.</p>
<p>To verify whether your version of OpenSSH supports PQC, run this command:</p>

</div>

    

</div>
<div class="prismjs aem-GridColumn aem-GridColumn--default--12"><figure>
  <pre data-cmp-is="prismjs" class="cmp-prismjs lang-bash"><code data-cmp-hook-prismjs="code" class="lang-bash">$ ssh -Q kex | egrep &#34;(mlkem|sntrup)&#34;
sntrup761x25519-sha512
sntrup761x25519-sha512@openssh.com
mlkem768x25519-sha256
$ </code></pre>
    <button data-cmp-hook-prismjs="copy" class="cmp-prismjs__copy">Copy</button>
  
</figure>


    

</div>
<div class="text bullet__xs padding__top-eight padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>Additionally, you will notice that the latest version of the OpenSSH client (version 10.1 and above) actively warns you if you are connecting to a server that does <i>not</i> use post-quantum key agreement:</p>

</div>

    

</div>
<div class="prismjs aem-GridColumn aem-GridColumn--default--12"><figure>
  <pre data-cmp-is="prismjs" class="cmp-prismjs lang-xml"><code data-cmp-hook-prismjs="code" class="lang-xml">$ ssh -V
OpenSSH_10.2p1, LibreSSL 3.3.6
$ ssh unsafe.example.com
** WARNING: connection is not using a post-quantum key exchange algorithm.
** This session may be vulnerable to &#34;store now, decrypt later&#34; attacks.
** The server may need to be upgraded. See https://openssh.com/pq.html
[...] </code></pre>
    <button data-cmp-hook-prismjs="copy" class="cmp-prismjs__copy">Copy</button>
  
</figure>


    

</div>
<div class="text bullet__xs padding__top-eight padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>This behavior is backwards compatible, meaning that clients without support for PQC will simply use classical key exchange algorithms, so updating your SSH deployments here should be reasonably painless.</p>

</div>

    

</div>
<div class="title padding__bottom-three title_sm aem-GridColumn aem-GridColumn--default--12">
<div class="cmp-title" id="title-21a9e4b22b">
    
      <h2 class="cmp-title__text" data-context="title">Virtual private networks (VPNs)<br />
</h2>
    
</div>

    


</div>
<div class="text bullet__xs padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>VPNs may use TLS to create their encrypted tunnels, and as such are able to take advantage of the common <span class="code">X25519MLKEM768</span> hybrid key exchange to make these tunnels quantum safe.</p>
<p>However, not all VPNs necessarily use TLS; instead, many implementations are based on Internet Protocol Security (IPsec), which uses the Internet Key Exchange (IKE). IKE is often based on a Diffie–Hellman key agreement and includes, for example, x509 client certificates for authentication. These mechanisms remain vulnerable to the quantum threat and thus require similar updates as are necessary for TLS.</p>
<p><a href="https://www.rfc-editor.org/rfc/rfc9370" target="_blank" rel="nofollow noreferrer">RFC9370</a> defines the means for IKEv2 to permit multiple key exchanges (and thus create <i>composite</i> keys). A <a href="https://datatracker.ietf.org/doc/draft-ietf-ipsecme-ikev2-mlkem/" target="_blank" rel="nofollow noreferrer">separate draft</a> specifies how to use ML-KEM as such an additional key exchange. This allows for the combination of a classic key with a post-quantum key, analogous to the hybrid PQ/T key exchange we are familiar with from TLS.</p>
<p>Different network equipment providers support PQC in IPSec VPNs to different degrees. Check with your vendor about their current post-quantum readiness in these products.</p>

</div>

    

</div>
<div class="title padding__bottom-three title_sm aem-GridColumn aem-GridColumn--default--12">
<div class="cmp-title" id="title-a0bb8a6134">
    
      <h2 class="cmp-title__text" data-context="title">Messages<br />
</h2>
    
</div>

    


</div>
<div class="text bullet__xs padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>Higher up on the application layer, one of the most widely used services — especially on mobile devices — includes text messages. This space is dominated by WhatsApp, Apple Messages, Signal, and any application or service using the Rich Communication Services (RCS) protocol, such as Google Messages.</p>
<p>Signal and WhatsApp both use the <a href="https://signal.org/docs/" target="_blank" rel="nofollow noreferrer">Signal Protocol</a>, which introduced the <a href="https://signal.org/blog/spqr/" target="_blank" rel="nofollow noreferrer">Sparse Post-Quantum Ratchet (SPQR)</a> in 2025. Cryptography experts broadly <a href="https://arstechnica.com/security/2025/10/why-signals-post-quantum-makeover-is-an-amazing-engineering-achievement/" target="_blank" rel="nofollow noreferrer">recognized</a> it as a well-designed engineering solution.</p>
<p>Likewise, Apple announced their <a href="https://security.apple.com/blog/imessage-pq3/" target="_blank" rel="nofollow noreferrer">post-quantum roadmap</a> for Apple Messages as far back as February 2024, introducing their &quot;PQ3&quot; post-quantum cryptographic protocol. This protocol only protects iMessages, however; messages sent using SMS or RCS (displayed on Apple devices in green, rather than blue, text bubbles) still rely on classic cryptography.</p>
<p>Google Messages offered (classic) <a href="https://blog.google/products-and-platforms/products/messages/helping-you-connect-around-world-messages/" target="_blank" rel="nofollow noreferrer">end-to-end encryption</a> as far back as 2020, but this was not standardized until the definition of the Messaging Layer Security (MLS) protocol in <a href="https://www.rfc-editor.org/rfc/rfc9420" target="_blank" rel="nofollow noreferrer">RFC9420</a>. Unsurprisingly, there are now <a href="https://datatracker.ietf.org/doc/draft-ietf-mls-pq-ciphersuites/" target="_blank" rel="nofollow noreferrer">efforts under way</a> to include ML-KEM and other post-quantum algorithms, which are included in the <a href="https://www.gsma.com/solutions-and-impact/technologies/networks/wp-content/uploads/2025/08/RCC.16-v2.0.pdf" target="_blank" rel="nofollow noreferrer">official specification</a>. We expect adoption of these to slowly pick up with the coming updates.</p>

</div>

    

</div>
<div class="title padding__bottom-three title_sm aem-GridColumn aem-GridColumn--default--12">
<div class="cmp-title" id="title-ab1004e42b">
    
      <h2 class="cmp-title__text" data-context="title">OpenPGP<br />
</h2>
    
</div>

    


</div>
<div class="text bullet__xs padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>OpenPGP, defined in <a href="https://www.rfc-editor.org/rfc/rfc9580.html" target="_blank" rel="nofollow noreferrer">RFC9580</a>, provides symmetric and asymmetric encryption and authentication via digital signatures. Despite many years of concerns and frustrations around the format itself and the usability of some of the most widely used implementations, PGP remains frequently used especially by open source projects and other software vendors to provide integrity and authenticity assurances of their software releases or security advisories.</p>
<p>Many organizations also use OpenPGP for encrypted communications over email. For example, Akamai allows security researchers to communicate with our security team using our <a href="/content/dam/site/en/documents/corporate/akamai-security-general.pub" target="_self">public PGP key</a> published on our <a href="/global-services/support/vulnerability-reporting" target="_self">Akamai Vulnerability Reporting</a> page.</p>
<p>As such, the OpenPGP standard is necessarily subject to the quantum threat and in need of updates to allow for the use of PQC algorithms. Within the IETF’s OpenPGP Working Group, a <a href="https://datatracker.ietf.org/doc/draft-ietf-openpgp-pqc/" target="_blank" rel="nofollow noreferrer">new draft</a> is being developed, which &quot;defines composite public key encryption based on ML-KEM (formerly CRYSTALS-Kyber), composite public key signatures based on ML-DSA (formerly CRYSTALS-Dilithium), both in combination with elliptic curve cryptography, and SLH-DSA (formerly SPHINCS+) as a standalone public key signature scheme.”</p>
<p>This draft has already seen <a href="https://sequoia-pgp.org/blog/2025/11/15/202511-post-quantum-cryptography/" target="_blank" rel="nofollow noreferrer">some adoption</a>, but as with many of the other cryptographic tools and libraries, you will need to closely track the official releases of the software you use to enable you to adopt and integrate new versions when they are published. It will likely still be a little while before we see widespread adoption of PQC in use for, say, software package signatures.</p>

</div>

    

</div>
<div class="title padding__bottom-three title_sm aem-GridColumn aem-GridColumn--default--12">
<div class="cmp-title" id="title-121e38f253">
    
      <h2 class="cmp-title__text" data-context="title">DNSSEC<br />
</h2>
    
</div>

    


</div>
<div class="text bullet__xs padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>Finally, we'll need to talk about the elephant in the room. Yes, it's DNS. (It's always DNS, isn't it?) Or, to be more specific, DNSSEC.</p>
<p>The Domain Name System Security Extensions (DNSSEC) provide cryptographic signatures on DNS resource records (RRs), primarily using (again) RSA and ECDSA as the signature algorithms. This, of course, means that these signatures are subject to the quantum threat, and any systems that rely on DNSSEC for integrity and authenticity assurance will need to adopt post-quantum cryptographic algorithms.</p>
<p>However, as we <a href="/blog/security/digital-signatures-traditional-post-quantum-cryptographic" target="_self">discussed previously</a>, there is a significant increase in the signing and verification performance, as well as in signature size, when moving from classic to PQC algorithms. The fact that DNS is primarily used over UDP exacerbates this problem compared with the HTTPS use case, as the larger signatures will no longer fit into a single UDP response packet.</p>
<p>This could lead to TCP fallback, packet fragmentation, slowed response times, and many other possible complications — not to mention the long, <a href="https://www.icann.org/en/public-comment/proceeding/proposed-root-ksk-algorithm-rollover-03-02-2026" target="_blank" rel="nofollow noreferrer">complex process</a> of algorithm rollover that will have to be coordinated starting with the root zone's Key Signing Key (KSK) and the possible need for multiple signatures across all signed zones during the transition period.</p>
<p>While the industry has not yet fully agreed on the right approach forward, a few proposals and <a href="https://datatracker.ietf.org/doc/draft-sheth-pqc-dnssec-strategy/" target="_blank" rel="nofollow noreferrer">strategic drafts</a> that are gaining traction involve the use of FN-DSA (i.e., what will become <a href="https://csrc.nist.gov/csrc/media/presentations/2025/fips-206-fn-dsa-(falcon)/images-media/fips_206-perlner_2.1.pdf" target="_blank" rel="nofollow noreferrer">FIPS 206</a>) or, alternatively, <a href="https://csrc.nist.gov/pubs/fips/205/final" target="_blank" rel="nofollow noreferrer">FIPS 205 Stateless Hash-Based DSA (SLH-DSA)</a> signatures in <a href="https://datatracker.ietf.org/doc/draft-harvey-cfrg-mtl-mode/" target="_blank" rel="nofollow noreferrer">Merkle tree ladder (MTL)</a> mode.</p>
<p>This approach would reduce the cost of signatures by avoiding the need to sign and verify each individual record in favor of signed Merkle “tree ladders,&quot; a concept similar to <a href="https://datatracker.ietf.org/doc/draft-davidben-tls-merkle-tree-certs/" target="_blank" rel="nofollow noreferrer">Merkle tree certificates</a> that are intended to help address the analogous concerns around multiple signatures in the TLS ecosystem.</p>
<p>Within the context of DNSSEC, the urgency of adopting PQC algorithms may not seem as immediate as in the case of protecting HTTPS traffic against the &quot;harvest now, decrypt later&quot; threat because the authenticity and integrity assurance of DNS traffic are not subject to a post-facto attack. However, the complexity of performing a<a href="https://www.icann.org/en/system/files/files/root-zone-algorithm-rollover-study-23may24-en.pdf" target="_blank" rel="nofollow noreferrer"> root KSK algorithm rollover</a> means that we need to plan for a prolonged timeline before all the DNS have been switched over.</p>
<p>Combined with the fact that, as noted above, we have not even agreed on what the final solution will look like, even a time horizon of a cryptographically relevant quantum computer and within the expected <a href="/blog/security/guide-international-post-quantum-cryptography-standards" target="_self">compliance requirements</a> from different jurisdictions means that we can't afford to delay that work.</p>

</div>

    

</div>
<div class="title padding__bottom-three title_sm aem-GridColumn aem-GridColumn--default--12">
<div class="cmp-title" id="title-b3add22229">
    
      <h2 class="cmp-title__text" data-context="title">Conclusion<br />
</h2>
    
</div>

    


</div>
<div class="text bullet__xs padding__bottom-thirteen aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" data-component="text" class="cmp-text">
    <p>Although Akamai is at the forefront of PQC adoption within the HTTPS ecosystem, having <a href="/blog/security/akamai-enables-post-quantum-cryptography-edge" target="_self">enabled</a> PQ/T hybrid key exchange for TLS 1.3, there is still a lot of work left to make all other systems and services quantum safe. The migration will be easier for some of the services we discussed above (e.g., SSH) but significantly more complicated for others (e.g., DNSSEC).</p>
<p>Systems that build on top of or use TLS make a migration easier, as most libraries and toolkits support PQ/T algorithms by now, and vendors have begun to offer these protections. However, the costs of keeping your cryptographic inventory up to date, identifying which components are ready for an update, and tracking the latest software releases remains significant.</p>
<p>For the other protocols, we recommend that you closely track the latest developments in the industry and to participate in the IETF and other standards discussions. As we continue to bring PQC to more of our own services, we will likewise continue to keep you updated in this blog series.</p>

</div>
