<script>
    document.addEventListener('DOMContentLoaded', function() {
        // Get all dropdown elements
        let dropdowns = document.querySelectorAll('.dropdown');

        // Add click event listener to each dropdown
        dropdowns.forEach(function(dropdown) {
            let dropdownName = dropdown.querySelector('.dropdown-name');
            let dropdownContent = dropdown.querySelector('.dropdown-content');

            dropdownName.addEventListener('click', handleFocus);
            dropdownName.addEventListener('mouseover', handleFocus);
            function handleFocus (event) {
                event.preventDefault();


                if (dropdownContent.style.display === 'none' || dropdownContent.style.display === '') {
                    dropdownContent.style.display = 'block';
                    dropdownName.classList.add('dropdown-active')
                }

                dropdowns.forEach(function(dropdown) {
                    let dropdownName = dropdown.querySelector('.dropdown-name');
                    let dropdownContent = dropdown.querySelector('.dropdown-content');
                    if (!dropdown.contains(event.target)) {
                        dropdownContent.style.display = 'none';
                        dropdownName.classList.remove('dropdown-active')
                    }
                });

            }
        });

        let isMouseInside = false;
        const navbar = document.querySelector('.navbar')

        navbar.addEventListener('mouseenter', () => {
            isMouseInside = true;
        });

        function handleClickOut(event) {
            if (isMouseInside) return;
            dropdowns.forEach(function(dropdown) {
                let dropdownName = dropdown.querySelector('.dropdown-name');
                let dropdownContent = dropdown.querySelector('.dropdown-content');
                if (!dropdown.contains(event?.target)) {
                    dropdownContent.style.display = 'none';
                    dropdownName.classList.remove('dropdown-active')
                }
            });
        }

        navbar.addEventListener('mouseleave', (event) => {
            setTimeout(() => {
                handleClickOut(event);
            }, 500)
            isMouseInside = false;
        });
        document.addEventListener("scroll", handleClickOut);
        document.addEventListener('click', handleClickOut);

    });
</script>

<p><strong>Quantum machine learning will not scale on claims alone. It must be benchmarked, reproducible, hardware-aware, and useful at every stage. With the release of MerLin 0.3 and our accompanying </strong><a href="https://arxiv.org/pdf/2602.11092"><strong>scientific publication</strong></a><strong>, Quandela introduces a discovery engine for photonic and hybrid quantum machine learning. Built on optimized strong simulation of linear optics and native PyTorch integration, MerLin enables systematic, proof-driven exploration of where quantum structure creates measurable value — and where it does not. Utility now. Scale next.</strong></p>



<p></p>



<h4 id='section-1'><strong>From isolated demonstrations to systematic discovery</strong></h4>



<p>Quantum machine learning (QML) has produced compelling ideas. But the field still faces a structural challenge:</p>



<p>Results are often:</p>



<ul class="wp-block-list">
<li>Difficult to reproduce</li>



<li>Evaluated under heterogeneous assumptions</li>



<li>Detached from hardware constraints</li>



<li>Reported without standardized baselines as it is usually done in the AI world</li>
</ul>



<p>If quantum computing is to become deployable — not just demonstrable — QML must evolve from isolated experiments to systematic benchmarking.</p>



<p>At Quandela, we believe in <strong>utility at every stage</strong> .<br>That principle guides both our hardware roadmap and our software ecosystem.</p>



<p>MerLin was built to bring that same discipline to quantum machine learning.</p>



<p></p>



<h4 id='section-2'><strong>What is MerLin?</strong></h4>



<p>MerLin (MerLin: A Discovery Engine for Photonic and Hybrid Quantum Machine Learning) is an open-source framework designed to:</p>



<ul class="wp-block-list">
<li>Integrate photonic quantum circuits directly into modern ML workflows</li>



<li>Enable scalable, differentiable training of quantum layers</li>



<li>Provide reproducible benchmarking pipelines</li>



<li>Remain hardware-aware by design</li>
</ul>



<p>It is built on <strong>Perceval</strong>, Quandela’s photonic quantum SDK , and leverages optimized Strong Linear Optical Simulation (SLOS) for exact simulation within the tractable regime.</p>



<p>Unlike fragmented tooling ecosystems, MerLin connects:</p>



<ul class="wp-block-list">
<li>Photonic circuit design</li>



<li>PyTorch-native model training</li>



<li>Benchmark-driven evaluation</li>



<li>Cloud and hardware execution pathways</li>



<li>Gate-based models to photonic models</li>
</ul>



<p>The objective is simple: <strong>turn photonic QML into an engineering discipline.</strong></p>

<h4 id='section-3'>Differentiable photonic quantum layers</h4>



<p>At the core of MerLin lies the QuantumLayer, a PyTorch-compatible module representing a parametrized linear-optical circuit.</p>



<p>Users can:</p>



<ul class="wp-block-list">
<li>Construct and manipulate photonic circuits using high level primitives without any need to understand the underlying physics</li>



<li>Encode classical data (angle or amplitude encoding)</li>



<li>Select measurement strategies (probabilities, expectations, amplitudes)</li>



<li>Train parameters through automatic differentiation</li>
</ul>



<p>MerLin accelerates simulation by precomputing sparse photon-number transition graphs. During optimization, only unitary-dependent coefficients are updated.</p>



<p>The result: photonic quantum circuits become trainable components within standard AI pipelines — without sacrificing physical realism.</p>



<p>This reflects a broader Quandela principle: <strong>Quantum must integrate into real computing environments, not remain isolated from them</strong>.</p>



<pre class="wp-block-code"><code>builder = CircuitBuilder(n_modes=6)
builder.add_entangling_layer(trainable=True, name="U1")
builder.add_angle_encoding(modes=list(range(6)))
builder.add_entangling_layer(trainable=True, name="U2")

layer = QuantumLayer(
    builder=builder,
    n_photons=3,      # 3 photons evenly distributed on 6 modes
    measurement_strategy=MeasurementStrategy.probs(),
)

optimizer = torch.optim.Adam(model.parameters(), lr=lr)
for _ in range(epochs):
    layer.train()
    optimizer.zero_grad()
    probs = layer(X_train)
    loss = F.cross_entropy(probs, y_train)
    loss.backward()
    optimizer.step()

# Your layer is now trained !</code></pre>

<h4 id='section-4'>Bridging photonic and qubit paradigms</h4>



<p>MerLin is not limited to photon-native architectures.</p>



<p>Through a QuantumBridge abstraction, qubit-based circuits can be mapped into photonic encodings (e.g., dual-rail representations). This enables:</p>



<ul class="wp-block-list">
<li>Cross-paradigm architectural comparison</li>



<li>Hardware-relevant translation of gate-based QML models</li>



<li>Controlled benchmarking under unified conditions</li>
</ul>



<p>Instead of debating abstractions, MerLin enables measurement.</p>



<p>This is aligned with our differentiation: <strong>Quantum that works — deployable, efficient, and built for scale</strong>.</p>



<p></p>



<h4 id='section-5'>Benchmarking 18 state-of-the-art models</h4>



<p>Alongside the framework, MerLin ships with <strong>18 reproduced state-of-the-art photonic and hybrid QML works</strong>, released as modular, reusable experiments.</p>



<p>This matters because QML results are still frequently reported under heterogeneous assumptions, with limited baselines and inconsistent reproducibility. MerLin’s reproduced-paper collection provides a shared empirical starting point — and a practical way to compare architectures under unified conditions.</p>



<p>Crucially, MerLin also supports cross-paradigm benchmarking: gate-based models can be translated into photonic encodings via the QuantumBridge, enabling controlled comparison between modalities.</p>



<p>Quantum software should not drift away from hardware realities. It should prepare for them.</p>



<p></p>



<hr class="wp-block-separator has-alpha-channel-opacity"/>



<h4 id='section-6'>Key technical insights</h4>



<p>Our reproduction campaign revealed several structural observations:</p>



<ul class="wp-block-list">
<li>Photon-native implementations can reproduce behaviors of gate-based architectures.</li>



<li>Encoding strategy significantly impacts robustness and expressivity.</li>



<li>Simulation efficiency gains enable larger-scale empirical exploration.</li>



<li>Not all reported quantum advantages remain under standardized baselines.</li>
</ul>



<p>These findings reinforce a simple idea:</p>



<p>Progress requires measurement.</p>



<hr class="wp-block-separator has-alpha-channel-opacity"/>



<h4 id='section-7'>Why MerLin matters</h4>



<p>Quandela’s purpose is to deliver photonic quantum systems that create value now and scale credibly toward fault tolerance.</p>



<p>MerLin extends that philosophy into quantum machine learning. It provides:</p>



<ul class="wp-block-list">
<li>A reproducible foundation for photonic QML</li>



<li>A bridge between classical AI and quantum hardware</li>



<li>A benchmarking culture grounded in evidence</li>



<li>An open ecosystem for researchers and developers</li>
</ul>



<p>Quantum computing will not mature through superlatives. It will mature through deployable systems, measurable progress, and disciplined engineering.</p>



<p>MerLin 0.3 is one step in that direction.</p>
